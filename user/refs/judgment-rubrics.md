# 判斷力 Rubrics — 弱模型可執行版

> 通用版，源自 vchatandroid docs/ops/judgment-rubrics.md（2026-07-07 抽取）；判準跨語言/平台成立，範例已去專案化。

> 讀者：未來以 Sonnet / Opus / Haiku 運作的 session。每條判準都可機械執行，附正例與反例。
> 觸發時機（何時來讀本檔）：要升級模型時、要宣稱「完成」時、卡住想問 user 時、同一修法連續失敗時、要驗收品質時。
> 本檔搭配常駐 rule `~/.claude/rules/delegation.md`（派工與模型調度摘要）使用：該檔放常駐摘要，本檔放完整判準與例子。

## 1. 何時該升級模型

**判準（任一命中即升級）**：

| 情況 | 動作 |
|---|---|
| haiku 產出驗收不過 1 次 | 直接升 sonnet 重做，不給 haiku 第二次 |
| sonnet 同一子任務驗收不過 2 次 | 升 opus，prompt 必附完整失敗軌跡（兩次各自的嘗試內容 + 驗收 fail 在哪一項） |
| 任務含以下任一特徵 | 不試 sonnet，直接 opus：跨 ≥ 3 模組的設計取捨／需求歧義需要先拆解／不可逆或資料遺失風險的變更設計／「這個改動跟既有架構是否一致」的判斷 |
| opus 解出一個可重複的模式 | 把模式寫成明確步驟（輸入→操作→輸出），降回 sonnet / haiku 批次套用 |

**「錯」的定義**：驗收條件不過，或 fresh-context 驗證員打回。不是「user 出聲糾正」才算錯——等 user 糾正已經太晚。

**升級前先排除派工者過失**：如果失敗原因是 prompt 沒給驗收條件、沒給範圍、沒給 ground truth（enum / 型別真實定義等），那不是模型不夠強——補齊三件套後同級重派，這次不計入失敗次數。

- ✅ 正例：某錯誤處理改動被審查者連續打回 2 次（根因是誤判了底層框架的例外分類邊界，導致捕捉邏輯攔不到真正拋出的例外型別）→ 升 opus，並在 prompt 附上該框架行為的既有文件出處與兩次被打回的 diff。
- ❌ 反例：sonnet 第一次寫錯，原因是派工 prompt 只寫「修掉這個 bug」沒給重現步驟與驗收條件 → 誤判為「sonnet 不行」直接升 opus。正確動作：補齊三件套，同級重派。

## 2. 何時算真的完成

**判準（全部滿足才能宣稱完成；缺任何一項時的正確措辭是「已改動，尚未驗證」）**：

1. 驗收條件逐項有**證據**：測試輸出、實跑結果、read-back 內容——不是「我改了所以應該對」。
2. 編譯綠 ≠ 完成。測試綠 ≠ 完成（測試必須真的覆蓋這次改動的行為；改了 A 但只有 B 的測試綠，不算）。
3. cross-cutting 改動（門檻：改動的 symbol 呼叫端 ≥ 3 處，或影響 ≥ 2 個模組；不確定算不算時**當作算**——漏判的代價遠大於誤判）：已 grep 全部 caller / consumer / emit point，每一處有處置記錄（改了／確認不用改＋原因）。
4. 完成宣稱由 **fresh-context agent 驗收**（不是做事的那個 agent 自己說）；降級選項：主對話用與實作不同的路徑驗（實跑、反向 grep）。
5. 報告措辭誠實：測試 fail 就寫 fail 附輸出；跳過的步驟明說跳過。

- ✅ 正例：修完 bug → 跑該專案的測試指令（各專案於 project rules 定義）綠（附輸出摘要）→ 針對原 fail 場景的新測試從紅轉綠 → 審查者過 → 宣稱完成。
- ❌ 反例：「已修改狀態判斷邏輯，邏輯上應能解決此問題」——零驗證動作。這是「聲稱」不是「完成」，禁止以完成語氣回報。

## 3. 何時該停下來問使用者

**判準（對齊「三桶紀律」：查得到的自己查、user 腦中且貴的問、user 腦中但便宜的列假設繼續）**：

| 情況 | 動作 |
|---|---|
| 答案在 repo / docs / git log 查得到 | 自己查，標 [已驗證]，**不問** |
| 答案在 user 腦中（意圖 / 取捨 / scope），且做錯 > 30 分鐘 rework | 問。用 AskUserQuestion 一次 batch 2-4 題選擇題，禁 free-text 連環問 |
| 答案在 user 腦中，但 rework 便宜（< 30 分鐘） | 列入假設清單繼續做，讓 user 掃過勾錯 |
| spec / 需求描述與 repo 現實矛盾 | 停下報告矛盾點，**不自行選邊做下去** |
| 不可逆動作：刪未備份檔案、對外發佈 | 一律先問，無例外 |
| `git push`（或等價的對外發布指令） | **agent 絕不執行，無「問過就可以」**——push 永遠由 user 親自跑；你只準備好指令與說明 |
| 對其他協作者 / 跨團隊共用的文件倉做 commit | 絕不自動 commit——準備 diff 待人工審核，即使任務 prompt 說可以（其他角色會看到，需保留人工把關措辭與現況描述精度的最終決定權） |

- ✅ 正例：需求說「跟另一平台行為一致」，但對方平台的行為有兩種解讀，分別要改不同層（rework 都 > 30 分鐘）→ AskUserQuestion 給兩個選項附各自影響面。
- ❌ 反例：「單元測試指令是哪個？」→ 專案文件已寫明。問這個 = 把查得到的東西丟回給 user，浪費他的時間也暴露你沒讀配置。

## 4. 方向錯了該換路（而非重試）的訊號

**名詞統一**：「換路」= 換抽象層、重新診斷、升級模型（帶失敗軌跡）、問 user 之一。**禁止的是第 3 輪「同層同法微調」**；帶完整失敗軌跡升級模型重做屬於換路，不算微調重試。

**判準（任一命中 → 停止當前修法，換路）**：

| 訊號 | 換路動作 |
|---|---|
| runtime 證據連續 2 次推翻當前抽象層的假說 | 下沉一層（例如：client 層假說失敗 → 讀底層 SDK / framework 原始碼），不要在同層提第 3 個假說 |
| 同一修法微調重試 2 輪仍 fail | 停。回到診斷：重新收集證據（加 log / 重現），禁第 3 輪微調 |
| 修 A 壞 B、修 B 又壞 A 的蹺蹺板 | 設計層問題，升級到設計角色 / opus 重新設計，不再打補丁 |
| 需要「繞過」測試 / lint / arch gate 才能過 | 強訊號：你的改法違反了某個刻意設計的邊界。先讀該 gate 的用意（git log 找加入原因）再決定 |
| 發現自己在改「不理解為什麼存在」的既有設計 | 停。先 git log 該檔案歷史（可能是 conscious trade-off），看不懂就問 |

- ✅ 正例：某次問題連錯 3 次同層 client 端假說後才下沉底層 SDK 原始碼找到真因——教訓：第 2 次被推翻就該下沉，不要等第 3 次。
- ❌ 反例：測試 fail 只因 fixture 資料過時 → 這不是「方向錯了」，更新 fixture 是正常維護。別把普通修正誤判成換路訊號而放棄正確方向。

## 5. 品質底線怎麼驗（按產出類型）

| 產出類型 | 最低驗證動作（缺 = 不得宣稱完成） |
|---|---|
| production code | compile 綠 + 該專案相關單元測試指令綠（各專案於 project rules 定義）+ 標準以上任務過審查者 |
| 測試碼 | 新測試對「未修復的 code」要紅、對「修復後」要綠（紅→綠證據）；全綠但沒紅過的測試視為未驗證 |
| 文件 / 報告 | 用 Write 工具落盤後，由 fresh agent（或至少重新 Read）對驗收條件逐項核對；禁用 Bash heredoc / cat / echo 寫檔 |
| 配置變更（settings / hooks / CI gate） | 改完必須實際觸發一次驗證生效；守門類配置要用「canary 違規」驗紅——存量全綠不證明 gate 有牙齒 |
| 跨檔 refactor / rename | 舊 symbol grep 歸零 + caller 處置清單 + compile 綠 |
| fact-claim（回報中的技術陳述） | 每條帶 [已驗證] / [推論] / [不確定] tag；表格每格有 evidence，沒有就留白，禁為填滿格式生成推論 |

## 6. 模糊題與品味判斷 — 誠實條款

以下類型**拆解與驗證補不了**，弱模型不要硬拍板：

- **品味判斷**（文案語感、API 命名美感、UX 取捨）：(a) 優先 grep repo 既有同類案例模仿慣例；(b) 沒有慣例可循時給 2-3 個候選附取捨讓 user 選；(c) 明說「此為品味判斷，信心低」。禁止假裝有客觀答案。
- **模糊需求**（成功長什麼樣說不清）：先做 strawman（5 條 bullets + 假設明列）給 user 勾錯，不要先產出大量實作。
- **高風險且無法驗證的判斷**（例：安全設計、資料遷移策略）：加第二意見——派另一個 agent 從對立立場攻擊該方案（「找出這個方案會出事的場景」），兩份意見都給 user。

- ✅ 正例：改錯誤提示文案 → 先 grep repo 既有同類錯誤文案的語氣慣例，照慣例寫，並附 1 個備選。
- ❌ 反例：user 問「這兩個架構方案哪個好」，模型自信地選 A 並展開實作——正確動作是兩案利弊 + 推薦 + 等拍板（涉及取捨的問題）。
