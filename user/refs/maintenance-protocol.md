# 制度檔維護協議（通用版）

> 通用版，源自 vchatandroid docs/ops/maintenance-protocol.md + letter-to-future-sessions.md 精華（2026-07-07 抽取）。

> 讀者：未來以任何模型運作的 session。本協議管的是「制度與配置檔」的修改，不管 production code。
> 制度檔泛指：專案根 CLAUDE.md、每 session 自動載入的 rules 檔、agent 定義檔、一次性 snapshot 診斷檔與制度文件目錄、跨 session 記憶索引檔與其 feedback/recent 條目。具體路徑依專案調整，本檔用類型描述而非寫死路徑。

## 1. 修改權限分級

| 級別 | 範圍 | 規則 |
|---|---|---|
| **自由改（不用問）** | 記憶系統新增條目、一次性事件記錄檔、索引檔加一行、制度文件目錄內容「補充」（加例子、加坑、修錯字）、審查/稽核報告類文件 | 照 §3 格式寫；記憶索引遵守 §4 上限 |
| **改前先備份，改後在回報中明示** | CLAUDE.md 局部更新（改既有 bullet 內容）、rules 檔內容更新、制度文件目錄的「規則語意變更」（不是補充而是改判準） | 先複製一份到當日備份目錄再改；回報必列「改了哪條規則、為什麼」 |
| **先問 user 才能動** | CLAUDE.md 結構重排或刪節、rules 檔增刪、agent 定義檔 frontmatter（model/tools 等執行參數）、settings 類設定檔、刪除任何制度檔、改「指令優先序」或「Single Source of Truth 表」 | 附 diff 預覽問過再動；user 沒回就不動 |
| **禁改** | 備份目錄**內容**（不得編輯既有備份檔；依 §4 輪替刪除整個過舊目錄不算違反）、一次性快照診斷檔（過時了就新寫一份，不改舊的） | — |

**特別提醒**：若專案的 settings 類設定檔被追蹤進版控（非預設慣例），改它等於改 shared config，一律落在「先問」級。

## 2. 每次踩坑後，教訓寫回哪裡

坑 = 任何「回頭看，如果早知道 X 就不會浪費這些時間」的時刻。當天寫，不要攢。

| 坑的類型 | 寫回目標 | 格式 |
|---|---|---|
| 第三方 SDK / framework 行為陷阱 | 專案既有的 SDK/framework 陷阱參考檔（沒有就新建一份，並在 CLAUDE.md 留一行索引） | 沿用該檔既有格式（現象/根因/修法） |
| user 糾正了工作方式 | 記憶系統新 `feedback_*.md` + 索引一行 | frontmatter + **Why:** + **How to apply:** |
| 派工失敗模式（subagent 答非所問、假 Read、擴 scope） | 派工模板文件對應章節的「失敗模式提醒」加一條 | 一行現象 + 一行防法 |
| 判斷錯誤（該問沒問、該停沒停、誤判完成） | 判斷準則文件對應節加正例或反例 | ✅/❌ 一行例子；判準本身要改 → 「先備份」級 |
| 專案架構/流程的新不變式 | root CLAUDE.md 對應節（一行）+ 細節放 reference 文件 | CLAUDE.md 只放一行 + 指向；寫回前先量測檔案大小，接近 §4 上限就改走抽檔路由 |
| 一次性事件記錄（修了什麼 bug） | 記憶系統 `recent_YYYY-MM-DD_*.md` | 不進制度檔——制度檔只收「可重複適用的規則」 |

**判斷不了寫哪的**：寫進記憶系統的 feedback 檔（最安全的預設），索引留一行。錯放的成本低，漏寫的成本高。

## 3. 寫入格式紀律

- 制度檔用 Write/Edit tool 寫，**禁 Bash heredoc / cat / echo 落盤**；寫完 Read 回驗。
- 規則寫法三要素：**觸發條件**（什麼時候適用）、**動作**（做什麼，可機械執行）、**例子**（至少一個）。抽象形容詞（「保持高品質」「小心處理」）視為沒寫，review 時刪除。
- 日期用絕對日期，不用「上週」「最近」。
- 事實陳述帶 [已驗證] / [推論] tag，引用附 檔案:行號 或 URL。

## 4. 膨脹控制 — 硬上限與精簡節奏

context tax 是這套制度的頭號敵人（見附錄 A）。上限到了就精簡，不是「找地方再塞一行」。以下數字為**預設值**，可依專案規模調整：

| 檔案 | 硬上限（`wc -c` bytes，預設值） | 到限動作 |
|---|---|---|
| root CLAUDE.md | 10,240 | 找出最情境性的段落抽到 reference 文件或制度文件目錄，留一行路由，屬「先問」級。**每次編輯後立即 `wc -c` 檢查**，不等月檢 |
| rules 目錄全部加總 | 18,432 | 同上；優先壓縮最少被用到的 rule；若要再降需砍內容，屬「先問」級 |
| 記憶索引「近期動態」 | 25 條 | size-triggered：**收到上限警告的當個 session 立即歸檔**（> 1 個月條目壓成一行搬歷史檔），不得拖延到「之後再說」。月檢發現已超過也可主動歸檔。為什麼是硬限：多數記憶系統自動載入只取檔案前段（例如「首 200 行或 25KB」）——超出部分不進 context，尾段內容會被靜默截斷 |
| 記憶索引單行 | 100 字 | 超了就把細節下沉到一次性事件記錄檔，索引只留一句 hook |
| 制度文件目錄各檔 | 12,288 | 例子太多時保留每判準最好的 1 正 1 反，其餘刪 |
| 備份目錄 | 保留最近 2 份 | 更舊的刪除（git 歷史裡都有，備份目錄只為快速回滾）。備份目錄不應進版控，如需要排除可加入 `.gitignore` |

量測指令（每月第一個 session 順手跑，路徑依專案調整）：

```bash
wc -c CLAUDE.md .claude/rules/*.md docs/ops/*.md
# 只數記憶索引「近期動態」段的條目（全檔 grep 會把其他索引也算進去，數字對不上上限）
awk '/^## 近期開發動態/,/^## 歷史動態/' <記憶索引檔路徑> | grep -c '^- \['
```

## 5. 更新後的自驗（不可省略）

1. Read 回改過的檔案，確認落盤內容 = 想寫的內容。
2. 改的是「判準/規則」時，用一個具體場景走一遍新規則，確認可執行（例：改了升級判準 → 拿一個真實失敗場景問自己新規則給什麼答案）。
3. 檢查有沒有製造新的重複：新內容若與既有檔案講同一件事，留一份，另一處改成指標（SoT 表見 root CLAUDE.md）。
4. 涉及計數/路徑的（「N 支 rules」「目錄下 N 個檔」）：寫之前用指令驗一次，或改寫成不含計數的措辭（計數必 drift）。

## 附錄 A：Context tax 是複利負債

每寫進「自動載入檔」（CLAUDE.md / rules / 記憶索引）的一行字，是對未來**每一個** session 收稅。這種負債是複利性的：不會因為某天沒動它而變輕，只會在沒人主動精簡時單向累加，直到某個 session 收到上限警告才被迫處理。應對原則：

- 記憶系統的歸檔依 size-triggered 政策執行：**收到上限警告的當個 session 立即做**，不拖延；定期盤點也可主動做（上限見 §4）。
- 想加一行常駐規則前先問：這條規則是不是每個 session 都用得到？用不到 → 放 on-demand 參考檔 + 一行路由，不放自動載入檔。
- 固定載入的稅源不只是自己寫的文字——外掛/工具/MCP 的說明清單也是每 session 全額付稅的一部分；若某項長期用不到，值得評估停用以降稅。

## 附錄 B：制度退化五模式 Checklist

這套制度最可能的退化方式，不是被推翻，而是被悄悄放棄執行——以下五種模式附早期訊號與預防矯正。

| 退化方式 | 早期訊號 | 預防 / 矯正 |
|---|---|---|
| **路由失效**：制度文件在參考目錄積灰，沒 session 去讀 | 連續多個 session 全程自己重新摸索、完成宣稱無驗證證據 | 常駐 rules 是防線；被使用者抓到違反時，矯正動作 = 重讀判斷準則對應節 + 把該情境補成正/反例 |
| **膨脹回彈**：規則又長回自動載入檔 | 量測指令跑出的數字超過 §4 硬上限 | §4 硬上限 + 每月量測指令，超了就抽檔留路由 |
| **儀式化**：模板照抄但驗收條件寫成「功能正常」這種不可證偽的話 | 驗收全 PASS 但使用者實測就壞 | 驗收條件必須可證偽（可觀察輸出/grep 結果/測試名）；「存量全綠」不證明 gate 有效，要主動製造一個違規案例驗證會被攔下 |
| **升降級被跳過**：同一子任務連續失敗仍在同層微調，不升級或換路 | 同一子任務第 3 次重試還在同層微調 | 次數判準是硬線：兩輪必換路（換抽象層 / 重新診斷 / 升級模型並附完整失敗軌跡 / 問 user）；派工時「執行者/model」欄必填 |
| **SoT drift**：規則改了 Single Source of Truth，別處的指標行沒同步或反向新增拷貝 | 兩份檔案對同一規則說法不同 | 只在 SoT 改；發現拷貝時把拷貝降級成指標 |
