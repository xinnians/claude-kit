# 派工 Prompt 模板 — 主對話直接套用

> 通用版，源自 vchatandroid docs/ops/delegation-templates.md（2026-07-07 抽取）；模板佔位符 {} 需依各專案填入。

> 讀者：未來擔任 PM 的主模型。用法：複製對應模板 → 填空（`{...}` 佔位，如 `{專案紅線：從該專案 CLAUDE.md / rules 摘入}`、`{專案測試指令}`、`{專案編譯指令}`；沒有對應內容就整段省略，不留空話）→ 派工。若專案有分角色的標準開發 pipeline（設計 → 實作 → 測試分工），**標準/複雜開發任務仍應走該 pipeline**，本檔用於 pipeline 之外的日常派工。

## 通用鐵律（每個模板都內建，改模板時不可刪）

1. **三件套必填**：目標與動機、驗收條件、回報格式。缺一件 = 派工品質問題，失敗不算模型的錯。
2. **回報合約**：subagent 只回結論 + `檔案:行號`；長產物用 Write 落檔、回傳路徑；禁止把整檔內容貼回主對話。
3. **ground truth 前置**：任務涉及 enum / sealed class / 函式簽名 / API 行為時，先 grep 真實定義塞進 prompt，不讓 subagent 憑印象（具名教訓 CHAT-4183：錯誤前提 fan-out 被 N 個 agent 一致背書）。
4. **model 顯式指定**：`model` 參數常見合法值 `sonnet` / `opus` / `haiku`（以當時 schema 為準）；不指定 = 繼承主對話模型，升降級時必須顯式指定。
5. **唯讀任務派唯讀 agent**（如 Explore、或專案的唯讀審查角色），防止調查員順手改檔。
6. **防偽引用機制**（模板 5 內建，其餘審查類派工也應套用）：驗收條件要求 agent「引用檔案中的具體內容」（如「引用該檔第一節標題」），空殼引用（answer 對但引不出原文）立刻穿幫——這是抓「假 Read / 假驗收」最有效的單一手段。

## 模板 1：搜尋 / 盤點（最常用）

**時機**：要回答「X 在哪裡 / 有哪些 / 誰呼叫了 Y」且預計要看 > 3 個檔案。
**agent**：`Explore`（唯讀）；機械枚舉可降 `haiku`，需判斷相關性用 `sonnet`。

```
subagent_type: Explore
model: sonnet        # 純枚舉可用 haiku；結果要做架構判斷時留給主對話，不叫 Explore 判
prompt: |
  [目標] 找出 {symbol/功能} 的 {全部定義處/全部 caller/全部 consumer}。
  [動機] 我接下來要 {改簽名/加分支/刪除}，需要完整影響面。
  [範圍] 搜尋 {目錄}；包含 {命名變體，如 XxxImpl、IXxx}。
  [搜尋策略] 至少兩種角度：(a) 精確 symbol grep；(b) {字串值/資源 id/反射} 等間接引用。
  [驗收條件] 每個命中點附 檔案:行號 + 一行說明該處怎麼用它；明說用了哪些 pattern、排除了哪些目錄。
  [回報格式] markdown 清單，≤ 60 行。找不到就回「0 命中 + 已試過的 pattern」，不要擴大解讀。
```

**失敗模式提醒**：Explore 回「應該在 X」而沒有行號 = 未驗證，退回重查；命中數異常少時先懷疑 pattern 拼錯而不是真的沒有。

## 模板 2：實作（小型、pipeline 之外的小改動）

**時機**：單檔、行為明確、rework 便宜的小改動。**標準/複雜任務走該專案的標準開發 pipeline，不用此模板。**
**前提**：仍須先過使用者的 plan 確認（多數專案要求動手前先過 plan），本模板不豁免確認。**agent**：`general-purpose`，model `sonnet`。

```
subagent_type: general-purpose
model: sonnet
prompt: |
  [目標] 在 {檔案路徑} 實作 {行為描述，含輸入→輸出}。
  [動機] {為什麼要改，一句話}。
  [ground truth] {貼上相關的既有簽名/enum 定義/被呼叫端的期望}。
  [範圍] 只改 {明確檔案清單}；不碰測試檔、不改其他 caller（發現需要改時停下回報，不擴 scope）。
  [專案紅線：從該專案 CLAUDE.md / rules 摘入]
  [驗收條件] {專案編譯指令} 綠（附輸出末 5 行）；相關單元測試綠（{專案測試指令}，附輸出）——
  無既有測試覆蓋時明確回報「無測試覆蓋」由主對話決定補測或豁免，不得默默略過；
  {行為級驗收，如：輸入 X 時回傳 Y 而非拋異常}。
  [回報格式] 改動摘要（每檔 1-2 行）+ diff 行數 + 驗收證據。不貼整檔。
```

**失敗模式提醒**：agent 遇到「無測試覆蓋」會默默跳過不回報——要求明確揭露；順手改了範圍外的 caller 視為擴大 scope，退回重派。

## 模板 3：重構 / 批次套用（模式已定、多點執行）

**時機**：主對話已解出修法模式，要在 N 個位置重複套用。
**agent**：`general-purpose`，model `sonnet`（模式極機械時 `haiku`）；多 agent 平行改檔須隔離工作區，沒有 worktree 類參數就序列執行。

```
subagent_type: general-purpose
model: sonnet
prompt: |
  [目標] 把以下已驗證的修法模式套用到 {N} 個位置。
  [模式定義 — 逐步照做，不要自行變體] 改前：{程式碼片段}；改後：{程式碼片段}；
  邊界情況：{哪種長相的不要改 + 例子}。
  [位置清單] {檔案:行號 列表 — 由主對話事先盤點，不讓執行者自己找}
  [驗收條件] 清單每項有處置記錄（已改/不適用+原因）；compile 綠；
  改完 grep {舊 pattern} 在範圍內歸零（附 grep 輸出）。
  [回報格式] 處置表（位置/動作/狀態）≤ 40 行。遇到不符合模式定義的位置：跳過並標記，不要即興發揮。
```

**失敗模式提醒**：執行者遇到「長得不太一樣」的位置會即興改——模板已要求跳過標記；主對話收到標記後自己判斷或升級。

## 模板 4：研究（外部技術 / 依賴行為 / 依賴升級）

**時機**：答案在 repo 之外（官方文件、changelog、SDK 源碼）。主對話禁止自己連環 WebSearch。
**agent**：`general-purpose`（可 WebSearch/WebFetch），model `sonnet`；多來源交叉驗證的大題目改用專案的深度研究類 skill（如有）代替本模板。

```
subagent_type: general-purpose
model: sonnet
prompt: |
  [目標] 查證 {具體問題，一題一個，可列 2-5 題}。
  [動機] 我要依答案決定 {決策}，答錯的代價是 {rework 描述} —— 所以寧可回「查不到」也不要編。
  [來源優先序] (1) 官方文件/源碼（{專案 SDK 源碼路徑，如有本機副本}）；(2) changelog/release notes；
  (3) issue tracker。部落格/論壇只能當線索，不能當結論。
  [驗收條件] 每個答案附出處（URL 或 檔案:行號）；出處必須真實存在；官方文件與源碼矛盾時兩者都列出。
  [回報格式] 逐題回答，每題 ≤ 6 行 + 出處。查不到就寫「查不到」+ 已試過的來源。
```

**失敗模式提醒**：agent 查不到答案時容易生成「聽起來合理」的來源頂替——出處必須可回頭驗證存在，不接受無出處的斷言。

## 模板 5：審查 / 驗收（驗證不自驗的執行載體）

**時機**：任何 agent（或主對話自己）宣稱完成後。**做事的 agent 不能當驗收的 agent。**
**agent**：code 審查優先用專案的唯讀審查角色（如有），否則 `general-purpose`；model `sonnet`；高風險升 `opus` 或派兩個獨立驗收比對。

```
subagent_type: general-purpose   # 或專案的唯讀審查角色（code 審查時）
model: sonnet
prompt: |
  [目標] 驗收以下產出，你是 fresh 視角，之前的實作討論你都沒看過——這是刻意的。
  [產出位置] {檔案路徑 / git diff 範圍，如 git diff HEAD~1}
  [驗收條件 — 逐項給 PASS/FAIL + 證據]
    1. {條件一，可觀察、可證偽}
    2. {條件二}
  [驗證方式] 檔案類實際 Read 全文核對（不是看摘要）；code 類實跑 {專案測試指令}；「已刪除/已歸零」類反向 grep 確認。
  [對抗性要求] 主動找碴：假設產出者想矇混過關，哪一項最可能是空殼？優先驗那項。
  [防偽引用] 至少一項驗收條件要求「引用檔案中的具體內容」（如引用該檔第一節標題），
  不是複述摘要——引不出來就是沒真的讀過。
  [回報格式] 逐項 PASS/FAIL 表 + 證據（輸出片段/行號）。FAIL 不需要給修法，定位問題即可。
```

**失敗模式提醒**：驗收 agent 也會假 Read（實案：agent 宣稱已讀檔案並驗證通過，實際內容與引用細節都是編的）。防法就是「防偽引用」——要求引用具體原文，空殼引用立刻穿幫。
